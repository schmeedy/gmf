Plug-in conventions/API conventions
  • Follows Eclipse naming conventions
    • Plugins are separated into .core and .ui for engine vs ui components
    • .ide for non-RCP supported plugins
    • Interfaces start with I
    • Anything that is in a package with .internal should be considered internal
    • All APIs have javadoc
    • Some APIs are provisional
    • JUnits are in .tests plugins

  • Domain dependencies are respected


  • .core plug-ins dont depend on .ui plugins
    • .ui plug-ins are optional

Diagram & Notation Domains: Benefits
  • Provides a logical persistence model for GEF through the presentation layer.
      • Defines a notation meta-model in EMF that is generic and logically separated from the semantic model persistence.  Clients of RMP-DL have diagram interoperability and compatible formats.
      • Allows utilization of additional services developed by RMP (common / MSL  undo / redo, query, copy/paste etc.)
      • Reuse of diagram code in different contexts.  Semantic model could be implemented through a bridge pattern.



  • Provide an extensibility mechanism for GEF concepts.
      • Support for on-demand loading for diagram components
      • Ease of maintenance through component based architecture.
      • Clients inherit extensibility mechanism that can be harnessed for their own individual domain extensions through partners.

Diagram & Notation Domains: EditPart Service
  • An Editpart provider creates an EditPart for a given model element.
  • Providers implement the IEditPartProvider with the following method:
    • EditPart createGraphicEditPart(View view)
  • Providers can provide new editparts for new or existing model element types.
  • Providers should avoid providing new editparts where the intent is just to customize the set of editpolicies (this can be done more efficiently using the EditPolicy service)
54
Diagram & Notation Domains: EditPolicy Service
  • An Editpolicy provider creates EditPolicies for a given editpart.
  • Providers implement the IEditPolicyProvider with the following method:
    • void createEditPolicies(EditPart editpart)
  • The service gives all interested providers a chance to create editpolicies for the given editpart in order of priority from lowest to highest.
  • Overriding an editpolicy is done by assigning a new one in the same role.
  • Assigning null to an editpolicy role is equivalent to removing that role behavior.
  • The service is very useful for adding/removing behavior without having to subclass targeted editparts.
55
Diagram & Notation Domains: Palette Service
  • Palette providers contribute to the palette of a given editor with a given content
  • Providers implement the IPaletteProvider with the following method:
    • void contributeToPalette(IEditorPart editor, Object content, PaletteRoot root)
  • Palette providers are encouraged to subclass the DefaultPaletteProvider, since all the contributions could be described in the XML extension.
  • Palette providers can contribute:
    • New Palette Drawers
    • New Palette Groups
    • New Palette Separators
    • New Palette Tool Items in drawers/groups.
    • New Palette Template Items in drawers/groups
56
Diagram & Notation Domains: Decoration Service
  • Decoration providers creates decorations for a given adapter.
  • Providers implement the IDecorationProvider with the following method:
    • void createDecorators(IDecoratorTarget decoratorTarget)
  • The IDecoratorTarget API lets the client install a custom Decoration or an image to a pre-defined location on the shape.
  • The service could be used by any client that needs to decorate a shape on the diagram without having any dependency to GEF.
57
Diagram & Notation Domains: Layout Service
  • Layout providers creates a Runnable object that will execute a layout for a given layout type.


  • Providers implement the ILayoutNodesProvider with the following method:
    • public Runnable layoutNodes( List layoutNodes, boolean offsetFromBoundingBox,
    • IAdaptable layoutHint);
    • The List is a list of Node shapes to be laid-out on the diagram surface.
    • The boolean offsetFromBoundingBox indicates if the layout is to be kept with-in the relative bounding box of the existing node shapes.
    • The layoutHint is an adaptable that usually adapts to String.  It tells the provider what kind of layout is desired.

EMF & OCL Domains: Model Integrity (cont.)
  • Extension point org.eclipse.emf.validation.constraintBindings
    • Allows clients of the EMF Validation framework to define "client contexts" that describe the objects that they are interested in validating, and to bind them to constraints that they are interested in enforcing on these objects
    • A client context can declare an <enablement> expression that matches model elements that are included in the context. Where that is not sufficient, an alternative is to define a selector class using a <selector> element.
    • Client contexts can be bound to constraints, individually, or to constraint categories (to bind all of the constraints in the category).
    • Binding to constraint categories has the advantage of allowing new constraint contributions in a category to automatically be bound to the appropriate client context, even if the constraint is defined in a plug-in that is unaware of that context or its binding to the category.
    • Category bindings are inherited by sub-categories from their ancestors.
65
EMF & OCL Domains: Model Integrity (cont.)
  • Extension point org.eclipse.emf.validation.validationListeners
    • This extension point is used to define validation listeners for the validation service (org.eclipse.emf.validation.service.ModelValidationService).
    • The validation service will inform this listener whenever validation has occurred, loading it if necessary in order to do so.
    • This is most useful for cases where client plug-ins need to find out about validation events even before they are loaded. Otherwise, it is usually simpler just to programmatically add a listener to the validation service.
    • The value of the <listener> element class attribute must be the fully qualified name of a class that implements the IValidationListener interface.
    • Listeners can also be registered in code, at run-time, using the ModelValidationService.addValidationListener() method.
    • In the next iteration work is being done to enhance this extension point to allow for registering listeners against specific client contexts
66
EMF & OCL Domains: Model Integrity (cont.)
  • The ModelValidationService singleton coordinates the invocation of validation.  It defines a single factory method for creation of IValidator for the batch and live evaluation modes.
  • Validators validate one or more objects at a time; the kind of object accepted as input depends on the evaluation mode.  They can be configured to report constraint passes as well as failures, for verbose results.
  • Results are reported as IValidationStatus.
  • Validators can be reused by a client for
    any number of validation operations.
  • The ILiveValidator validates EMF
    Notifications.
  • The IBatchValidator validates
    EObjects and, due to its support for
    model traversal, supports progress
    monitors.
  • Registered traversal strategies can
    be overridden by the client.

EMF & OCL Domains: Extensible Type Registry
  • Enables the creation of new types
    • Extensible framework for operations applicable to these types:
      • Creation
      • Modification
      • Destruction
    • Extensibility includes pre and post processing by third party plug-ins


  • These types extend the types defined for any given meta-model, so that IElementTypes can be used to distinguish types of elements that share the same metaclass.


  • Each element type defines how model objects that match its type are to be displayed (by icon URL and display name), as well as how they are to be created, modified and deleted (by edit helper and an accessor for edit commands).


  • There are two extensions of IElementType. These are
    • IMetamodelType
      • Each metamodel type maps directly to an EClass.
    • ISpecializationType.
      • Each specialization type matches a metamodel type with further restrictions expressed in a IElementMatcher class.

Common Domain: Common base plug-in classes
  • The common domain provides a base set of plug-in classes.
    • XtoolsPlugin - The abstract parent of a core plug-ins.
    • XtoolsUIPlugin - The abstract parent of a UI plug-ins.
  • Provides a standard template for xtools plug-in classes.
  • All xtools plug-ins extend one of these classes.
91
Common Domain: Common utility classes
  • The common domain provides shared utilities used in all xtools plug-ins.
    • AbstractResourceManager - I18N and resource management.
    • Log - A utility for logging errors, warnings, and information.
    • Trace - A utility for tracing debug information.
    • FileChangeManager  Handle resource change events.
    • FileModificationValidator  Determine if files are writable, check-out if required.
    • Shared widgets, dialogs, preference page utilities, etc.
92
Common Domain: Command infrastructure
  • Provides a set of classes for executing, undoing and redoing units of work, known as Commands .
  • This API is an implementation of the Command design pattern.
  • Using this API, application operations can be done, undone, redone and canceled without any dependency on UI components.
  • A command manager will maintain a history of the commands that it has executed that are eligible to be undone and redone.
  • Is currently independent of Eclipse but will be integrated with the Eclipse operations framework that is coming in Eclipse 3.1.
93
Common Domain: Service Providers Infrastructure
  • A service provider is an entity that provides some service (collection of related operations) to an interested client.
  • This concept is divided into:
    • A Service (from which clients request functionality).
    • Service Providers (which are the workhorses for the service being provided).
  • The common domain provides a set of services whose requests are fulfilled by service providers.
94
Common Domain: Services
  • Each service performs some specific piece of work for clients.
  • An operation represents the functionality provided by a service.
  • Operations are executed by passing them a provider to do the real work, based on the providers interface.
  • The Service uses execution strategies to select providers to service a client request.
    • FORWARD: The operation is executed by all providers in forward order.
    • FIRST: The operation is executed by the first provider.
  • Services also have performance optimization settings (caching).
  • Each service provides an Eclipse extension point so that providers can register with the service:
    • <extension-point id="iconProviders" name="Icon Providers" schema="schema/iconProviders.exsd"/>
95
Common Domain: Providers
  • Each service has one or more service providers.
  • Providers are register with the service by implementing the provider extension point.
  • The provider extension point specifies the provider and its priority.
  • Each provider registers a priority.
  • Prioritization provides a means of selecting one or more providers from a list of candidate providers based on their importance.
    • HIGHEST: signifies a highest ordered provider.
    • LOWEST: signifies a lowest ordered provider.
96
Common Domain: Editor Service
  • Provides an editor service for manipulating Eclipse editors.
    • Open an editor from an Eclipse editor input.
    • Provides methods to get registered editors
    • Register and unregister editors.
97
Common Domain: Icon Service
  • A service which provides icons.
  • The service is given an operation with a source object and icon options.
  • Based on the object and the provided options, an icon provider will return an icon.
    • Example: provide a icon for a logic element.
    • Example: provide an icon for a UML Element decorated with its stereotype image.
98
Common Domain: Parser Service
  • A service which provides parsed strings.
  • The service is given an operation with a source object and parser options.
  • Based on the object and the provided options, a parser provider will return a string.
    • Example: provide a name for the logic element.
    • Example: provide the fully qualified name for a UML Element.
    • Example: provide the edit string for a UML Operation contains the full signature.
99
Common Domain: Global Action Handler Service
  • This service provides an extensible way to handle commonly used (global) actions in different views and editors.
  • These global actions may appear in a view part context menu or may be predefined Eclipse retargetable actions.
  • The service allows different handlers to be used depending on the nature of the element(s) selected in the workbench part.
    •  Example: Override the global delete action when selecting an element on a diagram.
100
Common Domain: Action Filter Service
  • A service that extends what Eclipse provides for plug-in action enablement and visibility.
  • Provides a set of classes to help in evaluating expressions.
  • Allows clients to extend the action filters supported by objects selected in workbench.
  • It also permits the entire workbench selection to be considered so that an action can be enabled if it is supported on one or more elements in the selection.
    • Example: An action filter that will enable an action if the selected element can be duplicated.
    • Example: An action filter for elements that have a specific UML Stereotype applied.
101
Common Domain: Contribution Item Service
  • Provides a set of classes that facilitates the contribution of actions to Eclipse workbench parts.
  • It is meant to address some of the deficiencies in the Eclipse support for static action contributions, as follows:
    • Adds support for static contribution of custom contribution items
    • Adds support for static menu contributions to toolbar managers
    • Adds support for static contributions to menu managers that have been added programmatically to a toolbar
102
Common Domain: Properties Service
  • A set of services that allow clients to extends properties provided by a domain model.
  • An EMF model has a defined set of properties, clients wish to extend these properties.
  • Property providers allows a client to add additional properties to the domain.
    • Example: Allow a customer to add custom properties.
  • Property modifiers modify the domain properties.
    • Example: Override the default property modifiers to make them read only.
103
Common Domain: Marker Navigation Service
  • This service is the distribution point for "gotoMarker" requests made by Eclipse to an IEditorPart.
  • In xtools editors, the implementation of the gotoMarker() method simply forwards the request to this service.
  • Allows navigation to specific elements in diagrams.
    • Example: problem marker for an element, double click in the problems view goes to the highlighted element in the diagram.
104
Common Domain: Drag and Drop Services
  • Drag and Drop Listener Service
    • Register providers that provide drag source and drop target listeners against element types within workbench parts.
  • Transfer Adapter Service
    • Allows the definition of custom transfer agents to extend those available within Eclipse.

Example: Creating a Logic diagram editor (Cont.)
  • 1. Creating empty diagram plug-in
    • File New  Create new plug-in, RMP Diagram plug-in.
    • Specify the extension that the diagram editor resource is based on (.logic).


  • 2. Adding a new shape to the palette
    • Create a palette factory (LogicPaletteFactory)
    • Define an element type for the tool. (LogicType.ANDGATE)
    • Override createTool method in the palette factory and add a CreationTool entry with the type specified.
    • Define the extension in the plugin.xml.  Reuse the DefaultPaletteProvider but specify the new palette factory.
108
Example: Creating a Logic diagram editor (Cont.)
  • 3. Creating semantic model element for the shape.
    • Define element types in xml (could easily be generated based on wizard evaluation of semantic model).
    • Create the corresponding element helper class that manages the editing capability of the element.
    • Create commands to create elements if needed.  Default implementations exists for basic EMF creation.


  • 4. Creating the notation element for the shape.
    • Create a View Provider class that will create the view (LogicViewProvider)
    • Create an View factory class that will create the notation element and initialize it based on preferences and/or initial containment.  (LEDViewFactory)
    • In the View provider class, override the getNodeViewClass method to return the View class based on the hint provided (LogicConstants. TOOL_LED)
    • Define the View extension point and add the on-demand loading conditions.
109
Example: Creating a Logic diagram editor (Cont.)
  • 5. Creating the controller (EditPart) for the shape.
    • Create a EditPart Provider class that will create the editpart controller for the shape  (LogicEditPartProvider)
    • In the EditPart provider class, override the getNodeEditPartClass method to return the EditPart class name based on the View type passed into the method. (i.e. LEDEditPart)
    • Define the EditPart extension point and add the on-demand loading conditions.


  • 6. Displaying the shape on the diagram surface.
    • Create a new draw2d IFigure subclass that overrides the paintFigure routine to draw the figure as desired. (LEDFigure)
    • In the EditPart created in step 4, override the createNodeFigure routine to return the new Figure you just created..
